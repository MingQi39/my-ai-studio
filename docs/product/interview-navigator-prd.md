# 面试导航（Interview Navigator）产品需求文档

**状态：** 讨论稿（未实现）  
**版本：** v0.1  
**产品归属：** Qi AI Studio  
**核心方法：** KAN（Knowledge Atlas Navigator）

## 1. 一句话定义

面试导航是一个**由简历启动、用知识地图引导、以主动回忆为训练方式**的技术面试表达 Agent。

它不替用户批量编写或背诵面试答案；它帮助用户把真实经历和技术知识组织成可导航路径，使用户在被问到问题时能自己说清楚、经得起追问。

## 2. 问题与机会

当前求职者常见的困难不是“完全不知道”，而是：

- 简历写过、项目做过，但临场无法迅速定位可讲的内容；
- 能说出概念定义，却讲不出原理、取舍和真实工程证据；
- 写一份长答案成本高，背下来也无法应对变化的追问；
- 通用题库与个人经历脱节，练习收益低。

产品机会是把三类碎片连接起来：**简历中的项目事实、技术知识的关系、面试回答的组织路径**。

## 3. 产品原则

1. **地图优先**：任何训练先展示“当前问题处于哪张地图”，不从孤立问答开始。
2. **低输入**：用户上传简历、轻量确认、口头或短文本作答；不要求填写完整项目复盘或长答案。
3. **主动回忆优先**：默认先让用户回答；提示按需逐层露出，禁止默认给完整标准答案。
4. **真实证据优先**：只基于用户确认的简历/项目事实生成经历题；不知道就标记待补，而非编造。
5. **反馈定位断点**：反馈描述“缺失节点/断裂路径”，例如缺少 trade-off 或工程证据，而非笼统判定好坏。
6. **可复习**：每次训练沉淀为一张简短的导航卡，供间隔复习与再次口述。
7. **用户控制数据**：简历、解析事实和训练记忆均可查看、编辑、删除；跨项目长期记忆须显式开启。

## 4. 用户与首期场景

### 4.1 目标用户

- 准备中文技术面试的开发者；
- 有 1–3 个可讲项目、但缺乏系统表达训练的人；
- 希望将技术学习直接转为面试能力的人。

### 4.2 首期岗位范围

前端、全栈、后端、AI 应用工程相关岗位。首期侧重项目经历表达和用户简历中出现的技术专题。

### 4.3 首期不做

- 通用“代写简历”或海量题库；
- 自动虚构项目指标、职责、事故或成果；
- 英文面试、录音声学评分、真人面试匹配；
- 白板算法题、完整系统设计画板、招聘 JD 自动投递。

## 5. 产品地图

```text
                         求职目标
                    （岗位 / 级别 / 方向）
                              │
      ┌───────────────────────┼────────────────────────┐
      │                       │                        │
  知识地图                经历地图                  表达地图
  P5/P6/P7               项目/事实/证据             问题/回答路径
  概念与连接              决策/取舍/结果             30秒/2分钟/追问
      │                       │                        │
      └───────────────────────┴──────────┬─────────────┘
                                          │
                            [● 面试导航工作台]
                                          │
         选题 → 主动回忆 → 断点反馈 → 最小提示 → 重答 → 复习
```

### 5.1 三张关联地图

| 地图 | 节点 | 作用 |
| --- | --- | --- |
| 知识地图 | 例如 `HTTP → streaming → SSE → reconnect` | 展示概念的前置、关联与深度；标记 P5/P6/P7。 |
| 经历地图 | 项目、问题、职责、决策、约束、证据、结果 | 把用户可真实讲述的项目内容结构化。 |
| 表达地图 | Position、Principle、Mechanism、Trade-off、Evidence | 将题目转为回答时应经过的节点，而不是固定文稿。 |

## 6. 核心用户旅程

### 6.1 首次使用：三分钟建立首张地图

1. 用户上传 PDF 或 DOCX 简历。
2. 系统提取技术、项目和候选事实，并声明它们是“候选事实”。
3. 用户用低成本操作确认：`看起来对`、`删除`、`我不确定`，可选填写目标岗位。
4. 系统生成首张简历驱动的 KAN 面试地图，推荐一个 3 分钟训练路径。

示例确认界面：

```text
识别到：React · FastAPI · SSE · LLM · Docker
项目：Qi AI Studio · 旅行 Agent
目标方向：AI 应用工程 / 全栈（可修改）

[看起来对]  [删掉不相关项]  [选择目标岗位]
```

### 6.2 每日训练：一条小路径

```text
今日训练：SSE 流式聊天（预计 3 分钟）

HTTP → streaming → [SSE] → reconnect

Q：为什么这里选 SSE，而不是 WebSocket？
目标：走通“适用场景 → 取舍 → 你的项目证据”

[开始说 30 秒]  [给我一个提示]  [先看地图]  [跳过]
```

流程：

1. 先展示 Atlas 与当前节点；
2. 用户语音作答（后续）或输入短文本；
3. Agent 对照 Answer Route 识别已覆盖节点和缺失节点；
4. 只给缺失处的最小提示；
5. 用户立即用一句或一段重新组织；
6. 生成可复习的导航卡片。

### 6.3 项目经历训练

用户选择一个已确认项目后，系统展示经历地图，而非表单：

```text
Qi AI Studio
  用户问题 → 流式生成中断/刷新
       │
  方案候选：轮询 / SSE / WebSocket
       │
  [SSE] → 状态保存 → stream resume
       │
  取舍 → 风险 → 工程证据 → 结果 / 复盘
```

当用户讲不出某节点时，可点“我卡住了”。系统可选择展示：

1. 节点名称；
2. 一个回忆问题；
3. 关键词提示；
4. 仅在用户主动请求时，展示示例组织方式。

## 7. KAN 训练协议

每个技术专题遵循 KAN 的十阶段，但 MVP 以能训练为优先，不要求一次走完。

```text
① Position → ② Atlas → ③ Tree → ④ Principles → ⑤ Mechanism
      → ⑥ Connections → ⑦ Real Project → ⑧ Interview
      → ⑨ Compress → ⑩ Back to Atlas
```

### 7.1 面试题回答骨架

回答不保存为唯一标准稿，而保存为路线：

```text
Position → First Principles → Mechanism → Connections / Trade-off → Engineering Evidence
```

### 7.2 P5 / P6 / P7 题目分层

| 层级 | 训练目标 | SSE 示例 |
| --- | --- | --- |
| P5 | 定位与基础定义 | SSE 解决什么通信问题？ |
| P6 | 机制与取舍 | 为什么不是 WebSocket？断线如何恢复？ |
| P7 | 生产实践 | 代理缓冲、状态持久化、观测与容量如何处理？ |

### 7.3 断点反馈格式

```text
已走通：SSE → 单向服务端推送 → 重连
当前断点：Trade-off（为什么不是 WebSocket）
最小提示：比较通信方向、交互复杂度与维护成本。
下一步：用一句话补上这个节点。
```

禁止默认输出整篇参考答案。

## 8. 功能范围与优先级

### P0：可用闭环

| 功能 | 说明 | 验收结果 |
| --- | --- | --- |
| 简历上传与解析 | 支持 PDF/DOCX；提取候选项目和技术节点 | 用户可一键确认/删除提取结果。 |
| 面试地图 | 基于确认内容生成知识/经历/表达关系 | 用户能看见当前问题的位置和关联节点。 |
| 单题训练 | 短文本作答、提示、重答 | 系统能指出已覆盖/缺失节点。 |
| 题目分层 | P5/P6/P7 与经历题 | 用户可从同一主题逐层练习。 |
| 复习卡 | 保存当前路径、断点、下一题 | 用户下次能从断点继续。 |

### P1：提高完成率

- 浏览器语音输入和转写；
- 基于岗位 JD 的题目权重调整；
- 连续追问树；
- 项目仓库连接，生成“候选工程证据”并让用户确认；
- 每日 3 分钟复习计划。

### P2：后续探索

- 英文面试表达；
- 面试录音节奏与口头禅反馈；
- 多轮模拟面试报告；
- 多个求职方向的独立地图。

## 9. 信息架构与界面

### 9.1 页面

1. **入口/导入页**：上传简历、确认提取信息、选择岗位方向。
2. **训练主页**：今日推荐、继续练习、复习卡片。
3. **面试导航工作台**：地图、当前题、作答与反馈。
4. **项目经历页**：项目的候选事实与经历地图。
5. **复习页**：按断点和间隔复习安排题目。
6. **数据与隐私页**：简历、解析内容、学习状态的查看/删除。

### 9.2 工作台布局

```text
┌──────────────┬───────────────────────────────┬──────────────────┐
│ 面试地图       │ 当前练习                        │ 反馈与下一步       │
│              │                               │                  │
│ ▾ 项目表达    │ Q：为什么选 SSE？                │ 已覆盖：          │
│   ● SSE       │                               │ ✓ Position        │
│   ○ reconnect │ [短文本/语音转写区]              │ ✓ Mechanism       │
│              │                               │ ○ Trade-off       │
│ ▾ 技术专题    │ [回答] [提示] [看地图]            │ → 补一个节点再答   │
│ ▾ 复习        │                               │                  │
└──────────────┴───────────────────────────────┴──────────────────┘
```

## 10. 数据模型（概念层）

```text
InterviewProfile
 ├─ ResumeDocument
 ├─ ResumeClaim[]             # 解析的候选事实，必须用户确认
 ├─ InterviewGoal[]           # 岗位、级别、重点方向
 ├─ KnowledgeNode[]           # 概念、关系、P5/P6/P7、学习状态
 ├─ ExperienceNode[]          # 项目、决策、证据、结果、确定性
 ├─ AnswerRoute[]             # 问题 → 目标节点 → 已覆盖/缺失节点
 └─ ReviewCard[]              # 压缩卡、断点、下次复习时间
```

### 10.1 关键约束

- `ResumeClaim` 初始状态为 `candidate`，只有用户确认后才能作为经历题依据；
- `ExperienceNode` 中的职责、指标与结果不得由模型无依据生成；
- 回答原文保留版本，Agent 的建议与用户原话分开存储；
- `KnowledgeNode` 可脱离简历存在，但经历题必须可追溯到确认的事实；
- 所有生成均写入 trace：使用的节点、提示词版本、模型与时间。

## 11. 隐私与安全

简历包含个人信息，MVP 必须满足：

1. 上传前明确说明用途：仅用于生成个人训练地图与题目；
2. 提供删除简历原件、解析数据、训练记录的入口；
3. 默认不将简历用于跨用户训练，也不自动公开；
4. 日志、错误追踪与模型请求中避免不必要地保留联系方式、身份证明等敏感字段；
5. 用户可以关闭跨项目偏好记忆；
6. 当模型无法确认事实来源时，标注“不确定”，不能补写。

正式实现前需要确认文件存储、保留期、模型供应商数据政策与用户授权文案。

## 12. 成功指标

### 北极星指标

**完成一次“回答 → 断点提示 → 重答”闭环的周活跃用户占比。**

### MVP 指标

- 简历上传后完成候选事实确认的比例；
- 首日完成至少 1 次训练闭环的比例；
- 单次训练平均输入字符数/语音时长（越低输入、越高完成越好）；
- 重答后 Answer Route 覆盖节点的提升；
- 7 日内完成复习卡回访的比例；
- 用户主动标记“题目与我的经历相关”的比例。

不以“生成答案字数”或“模型输出长度”作为成功指标。

## 13. MVP 验收场景

### 场景 A：简历到首个训练

给定一份包含 React、FastAPI 和 AI 项目的简历，用户能在不写长文本的前提下：

1. 上传并确认识别出的技术/项目；
2. 看到包含至少一个技术节点、一个经历节点和一个表达路径的地图；
3. 在 3 分钟内完成一题；
4. 获得明确的缺失节点和一次重答机会；
5. 保存一张下次可继续的复习卡。

### 场景 B：缺少事实时不编造

当简历只写“参与流式聊天功能”，没有职责或结果时，系统可提问或标记待确认，但不得声称用户“设计了重连架构”或取得具体指标。

### 场景 C：题目与地图保持连接

当用户练习 “SSE vs WebSocket” 时，界面应能显示该题位于 `HTTP → streaming → SSE`，并能链接到对应的项目经历节点；不能只显示孤立题干和范文。

## 14. 关键产品决策（待确认）

1. **已确认：** 本地优先提取 PDF/DOCX 的结构与关键词；模型仅接收用户确认后的最小必要关键词（技术栈、项目名、角色/年限、项目要点），不自动发送简历原文、联系方式或完整文件。
2. MVP 的首个输入是否只支持 PDF，还是同时支持 DOCX？
3. 首期是否开放语音输入；若不开，短文本回答是否足够满足“低输入”？
4. 是否将该能力作为现有聊天中的 `面试导航` 专用会话类型，还是独立工作台？建议：**专用会话类型 + 独立工作台布局**，复用认证、BYOK、SSE 和会话基础设施。
5. 首期是否允许连接用户代码仓库作为候选工程证据？建议：后置到 P1，先通过简历和用户确认验证训练闭环。
